## 1. 왜 리모델링하는가

### 1.1 현재 상태의 문제

프로필 탭에 성격이 다른 기능 5가지가 평평하게 섞여 있음:

1. 페르소나 정보 (읽기)
2. 퀵 토글 — 감기/시술/야외 (Tier 3 임시 상태)
3. 건강 프로필 수정 진입 (쓰기, 드물게)
4. 알림 설정 진입 (앱 동작 설정)
5. 기준치 숫자 요약

유저 관점에서 "지금 뭐 하러 왔지?"가 명확하지 않음. 매일 들어올 이유도, 가끔 들어올 이유도 불분명.

### 1.2 프로필 탭의 본질 (재정의)

**진짜 상품은 "알림"이다.** 유저가 앱을 안 켜도 알림은 돌아가야 한다.

프로필 탭은 알림이 **정확**해지기 위해 앱이 "오늘의 너"를 알게 하는 공간이다. 그런데 유저는 프로필 업데이트할 동기가 약하다. 안 고쳐도 당장 앱이 안 죽기 때문.

→ 그래서 프로필 탭은 **유저에게 숙제를 주면 안 된다.** 최소한의 존재감만 있어야 한다.

### 1.3 프로필 탭의 역할 (두 가지)

1. **"앱이 나를 이렇게 보고 있구나"**를 3초 안에 확인시켜주기 (신뢰감)
2. **"뭔가 달라졌을 때만 빠르게 반영할 수 있게"** 열어두기 (접근성)

그 이상은 요구하지 않는다. **앱이 잘한 걸 자랑하는 공간이 아니라, 유저가 알고 싶을 때 준비돼 있는 공간.**

## 2. 전체 구조 (3단 깊이)

```
[1단] 프로필 탭
   │
   ├─ 페르소나 카드 (확장형)
   │    └─ [자세히 보기 ˅] 탭하면 같은 카드 내에서 아래로 확장
   │         (페이지 전환 없음)
   │
   └─ 내 몸 정보 링크 ──→ [2단] 내 몸 정보 화면
                              (별도 라우트, 요약 편집 구조)
                              각 항목 탭 → 바텀시트로 수정

[우측 상단 설정 아이콘] ──→ [2단] 설정 화면 (신규)
                              알림 설정 / 재진단 / 앱 정보
```

### 원칙

- **얕게 들어온 유저는 1단만 보고 나간다.**
- **궁금한 유저는 2단을, 수정할 유저는 3단까지 내려간다.**
- 아래로 갈수록 **편집 밀도가 높아지고**, 위는 **읽기 중심**.

- 측정소 위치 표시 (프로필 탭에 불필요)

## 4. 2단: 내 몸 정보 화면 (별도 라우트)

### 4.1 레이아웃

```
┌─────────────────────────────────┐
│  ← 내 몸 정보                  │
├─────────────────────────────────┤
│                                 │
│  기본 정보                      │
│  ─ 닉네임         yulgok  >    │
│  ─ 출생 연도      1990    >    │
│  ─ 성별           남성    >    │
│                                 │
│  건강 상태                      │
│  ─ 호흡기         건강해요 >   │
│  ─ 민감도         매우 예민 >  │
│  ─ 야외 활동      3시간+  >    │
│  ─ 활동 유형      운동·통근 >  │   ← §Q3 보완
│                                 │
│  현재 상황                      │
│  ─ 임신         해당 없음 >    │
│  ─ 피부 시술    해당 없음 >    │
│                                 │
└─────────────────────────────────┘
```

> **§Q7**: 알레르기 시즌 항목 제외. "현재 상황" 섹션은 임신·피부 시술 2개만. `pollenAllergySeason` 은 출시 후 추가.

### 4.2 기존 필드 처리 (§Q3 보완)

- **`activityTags`** (통근/산책/운동/배달/육아): 건강 상태 섹션에 "활동 유형" 항목으로 추가. 바텀시트에서 다중 선택.
- **`discomfortLevel`** (현재 불편 정도 0/1/2): 화면에서 **표시 안 함**. 필드는 유지 (다른 코드가 쓸 가능성).
- **`skinTreatmentDate`** (시술 날짜, 14일 만료 계산용): 별도 항목 아님. "피부 시술" 바텀시트 내부에서 날짜 선택 UI로 처리.

### 4.3 동작

- 각 항목 탭 → **바텀시트**로 해당 항목만 수정
- 전체 온보딩 플로우 재진행 없음
- 현재 값이 보이고 빠르게 수정 가능
- 맨 아래 "재진단" 버튼 **없음** (→ 설정 화면으로 이동)

### 4.4 의도

수정할 내용이 많고 편집 맥락이 필요한 작업이라 별도 화면.
카드 확장이 아닌 페이지 전환으로 간 이유: 읽기 ≠ 쓰기.

### 5.2 이동되는 것

- **알림 설정**: 기존 프로필 탭에 있던 진입점 → 설정으로 이동 (방해 금지 포함)
- **재진단**: 기존 "내 정보" 화면에 있던 버튼 → 설정으로 이동
- **앱 정보/개인정보처리방침**: 기존 위치 확인 후 설정으로 통합
- **`QuietHoursSection`** (기존 `/profile/edit`): `/notifications` 화면 내부에 이미 방해 금지가 있으므로 별도 이동 없음. 단계 5에서 `/profile/edit` 삭제 시 자연스럽게 제거됨.

### 5.3 의도

프로필 ≠ 설정. 프로필은 "나에 대한 기록", 설정은 "앱 동작 조정".
성격이 다르므로 분리.

## 6. 페르소나 시스템 개편

### 6.1 이름 정비 (6종)

| 기존 (persona_generator.dart) | **새 이름** | 조건 |
|--|--|--|
| 복합 주의형 | **세심한 케어형** | 호흡기 + (야외 OR 체감) |
| 조용한 취약형 | **섬세한 체질형** | 호흡기만 |
| 활동 민감형 | **활발한 감지형** | 야외 + 체감 |
| 야외 활동형 | **야외 라이프형** | 야외만 |
| 체감 민감형 | **예민한 감지형** | 체감만 |
| 일반 관리형 | **균형 유지형** | 해당 없음 |

### 6.2 서브타이틀

| 페르소나 | 서브타이틀 |
|--|--|
| 세심한 케어형 | 건강과 일상 둘 다 챙기는 당신에게 맞춘 기준이에요 |
| 섬세한 체질형 | 증상이 없어도 먼저 반응하는 체질을 고려했어요 |
| 활발한 감지형 | 많이 움직이고 빨리 감지하는 당신을 위한 기준이에요 |
| 야외 라이프형 | 바깥 시간이 많은 당신에게 맞는 기준이에요 |
| 예민한 감지형 | 공기 변화를 빨리 알아차리는 감각을 반영했어요 |
| 균형 유지형 | 지금 상태라면 기본 기준으로 충분해요 |

### 6.3 근거 포함 규칙 (§Q1 확정)

**규칙**

- **페르소나가 `general`(균형 유지형)이면 `reasons = []`. 무조건 빈 리스트.**
- **다른 모든 페르소나의 경우, 값이 0이 아닌 조건 전부를 근거에 포함.**
  (분류 임계값 기여도와 상관없음)

**이유**

유저 관점에서 "내가 답한 내용은 모두 반영되어야" 투명성이 느껴진다.
분류 가중치는 앱 내부 판단이고, 유저가 답한 데이터는 모두 "당신이 이렇게 알려줬어요"로 표시되어야 한다.

**예시**

- `respiratoryStatus=1 + outdoorMinutes=1 + sensitivityLevel=0` → 페르소나 `compound` 분류됨 → 근거에 비염, 야외 1~3h 둘 다 포함 (sensitivityLevel=0은 제외)
- `respiratoryStatus=0 + outdoorMinutes=0 + sensitivityLevel=1` → 페르소나 `general` → 근거 빈 리스트

### 6.4 근거 문구 풀 (D 톤: 공감 + 팩트)

**톤 원칙**
- 제목은 간결 (명사형)
- 설명은 "당신이 답한 내용"을 존중하는 방식
- 끝은 "그래서 기준을 낮췄어요" 또는 "그 감각을 반영했어요"로 투명성 전달

**호흡기** (`respiratoryStatus`)
| 값 | 제목 | 설명 |
|--|--|--|
| 0 | (근거 없음) | — |
| 1 (비염) | 비염 | 코 점막이 먼지에 민감해요. 그래서 기준을 낮췄어요. |
| 2 (천식) | 천식 | 적은 농도에도 기관지가 반응해요. 그래서 기준을 낮췄어요. |
| 3 (둘 다) | 비염과 천식 | 코와 기관지가 먼지에 이중으로 반응해요. 그래서 기준을 낮췄어요. |

**야외 활동** (`outdoorMinutes`)
| 값 | 제목 | 설명 |
|--|--|--|
| 0 | (근거 없음) | — |
| 1 | 하루 1~3시간 야외 | 노출 시간이 있는 편이라 평균보다 조금 더 영향을 받아요. |
| 2 | 하루 3시간 이상 야외 | 노출 시간이 길어서 같은 농도여도 영향이 커요. |

**체감 민감도** (`sensitivityLevel`)
| 값 | 제목 | 설명 |
|--|--|--|
| 0 | (근거 없음) | — |
| 1 | 조금 예민한 체질 | 평소 미세먼지 변화를 느끼는 편이잖아요. 그 감각을 반영했어요. |
| 2 | 매우 예민한 체질 | 수치가 낮아도 몸이 먼저 알아채잖아요. 그 감각을 반영했어요. |

**Tier 2 기간 상태** (§Q2 범위 제한)

이번 리모델링 범위에 **포함**되는 것 (UserProfile에 직접 필드로 존재):

| 항목 | 제목 | 설명 |
|--|--|--|
| `isPregnant == true` | 임신 중이세요 | 태아에게 미세먼지가 닿지 않도록 더 조심스럽게 설정했어요. |
| `recentSkinTreatment == true` | 피부 시술 회복 중 | 자극을 피해야 하는 시기잖아요. 그래서 기준을 더 낮췄어요. |

이번 리모델링 범위에서 **제외**되는 것 (출시 후 추가 예정):

- `withInfant` (영유아 동반)
- `immunoSuppressed` (면역저하/항암)
- `pollenAllergySeason` (꽃가루 시즌)

이 세 항목은 `TemporaryStateType` enum으로는 정의되어 있으나, `UserProfile` 직접 필드가 아니라 `List<TemporaryState>` 로 관리됨. 근거 포함 시 `PersonaGenerator.generate()` 시그니처가 변경되어 호출처 전부 수정 필요 → 출시 후 개선.

### 6.5 Persona 클래스 스키마 변경

```dart
class Persona {
  // 기존 필드는 유지 (keywords, actionGuides 등 — 다른 화면에서 사용 중일 수 있음)

  // 신규 필드
  final List<ReasonItem> reasons;
}

class ReasonItem {
  final String title;        // 예: '천식'
  final String description;  // 예: '적은 농도에도 기관지가 반응해요...'
  const ReasonItem({required this.title, required this.description});
}
```

- `general` 페르소나는 `reasons = []` 반환
- `PersonaGenerator._buildReasons(UserProfile)` 메서드에서 조건별로 리스트 생성
- 시그니처는 기존과 동일 (`UserProfile` 하나만 받음). Tier 2는 `isPregnant`, `recentSkinTreatment` 만 직접 참조.

- 매일 쓸 동기가 약하고, 안 쓰면 프로필이 오염됨
- 유지 비용 > 실제 가치
- 출시 후 사용자 피드백으로 진짜 필요한지 재판단 후 추가

### 7.3 대체 동선

유저가 "오늘 몸이 안 좋아"를 앱에 알리고 싶다면:
→ 프로필 탭 → 내 몸 정보 → 체감 민감도 수정 (바텀시트)

번거롭지만 **자주 있는 일이 아님**. 매일의 숙제를 없앤 대가.

## 8. 기존 코드 처리 (§충돌 반영)

### 8.1 기존 `/profile/edit` 라우트 (§충돌2)

- **단계 5에서 삭제 + `/my-body-info` 로 리다이렉트** (§Q5: 리다이렉트는 설정 화면 완성 후)
- 기존 화면의 섹션별 처리:
  - `DiagnosisBanner` (재진단 버튼) → 설정 화면으로 이동
  - `TodaySituationSection` (퀵 토글 Tier 3) → 전체 삭제 (§7)
  - `TemporaryStatesSection` (Tier 2) → "내 몸 정보" 화면 "현재 상황" 섹션으로 흡수
  - `BasicInfoSection` (Tier 1) → "내 몸 정보" 화면 "기본 정보" + "건강 상태" 섹션으로 흡수
  - `QuietHoursSection` → **건드리지 않음**. `/notifications` 화면 내부에 방해 금지가 이미 존재하므로 별도 이동 없이 단계 5 삭제 시 자연 제거 (§Q6)

### 8.2 CareTab → `/profile/edit` 직접 링크 (§충돌1)

- 단계 5에서 `/my-body-info` 로 변경
- 다른 feature 건드리는 작업이므로 설계 §10.3 규칙에 따라 단계 5 진입 전 최종 확인 필수

### 8.3 페르소나 이름 문자열 직접 비교 (§충돌3)

현재 `profile_tab.dart`에 존재하는 패턴:

```dart
String get _emoji => switch (persona) {
  '복합 고위험군' => '🚨',
  '호흡기 취약형' => '🫁',
  '임산부 보호형' => '🤰',
  '민감형 관리자' => '🎯',
  '활동형 아웃도어' => '🏃',
  _              => '🛡️',
};
```

**현재 이미 깨진 상태**: `persona_generator.dart` 반환값("복합 주의형" 등)과 매치 안 돼서 default `🛡️` 반환됨.

**처리**: 문자열 비교를 **`PersonaType` enum 비교**로 전환. 이모지/색상 매핑을 `Persona` 객체의 `emoji`, `type` 필드로 직접 참조. 이 작업은 **단계 1에 포함**.

### 8.4 단계 1 완료 후 표시 불일치 (§충돌4)

- 단계 1 완료 시점에서 UI는 새 이름("세심한 케어형" 등)을 표시하지만, `reasons` 필드는 UI에 연결 안 됨 (단계 4에서 연결).
- 이건 의도된 중간 상태. Claude Code 확인만 필요.

## 9. 시공 단계 (5단계)

안전한 순서. **각 단계는 독립적으로 앱이 작동하는 상태**여야 함.

### 단계 1: 페르소나 시스템 개편
- `persona_generator.dart` 개편 (이름, 서브타이틀, `reasons` 필드 추가)
- `_buildReasons()` 메서드 구현 (§6.3 규칙)
- `profile_tab.dart` 내 문자열 비교 로직 → enum 비교로 전환 (§8.3)
- 테스트 추가
- 그 외 UI 파일은 건드리지 않음

### 단계 2: 설정 화면 껍데기 생성
- 새 라우트 `/settings` 만들기
- 빈 설정 화면 (알림 / 진단 / 앱 정보 섹션만 있고 아이템은 아직 없음)
- 라우팅 연결만 확인

### 단계 3: 내 몸 정보 화면 (별도 라우트)
- 새 라우트 `/my-body-info` 생성
- 기본 정보 / 건강 상태 / 현재 상황 3섹션
- 각 항목별 바텀시트 수정 위젯
- `activityTags` 항목 추가 (§Q3)
- `discomfortLevel` 표시 안 함 (§Q3)
- `skinTreatmentDate` 는 피부 시술 바텀시트 내부 처리 (§Q3)
- "현재 상황" 섹션: 임신·피부 시술 2개만 (알레르기 시즌 제외, §Q7)
- 재진단 버튼 없음
- **`/profile/edit` 리다이렉트 추가하지 않음** (§Q5: 단계 5로 이동)

### 단계 4: 프로필 탭 새 페르소나 카드
- 확장형 페르소나 카드 (자세히 보기 ↔ 접기)
- 기준치 비교 표시
- "왜 더 엄격한가요" 근거 리스트 (단계 1의 `reasons` 사용)
- 균형 유지형은 근거 블록 대신 안내 문구

### 단계 5: 퀵 토글 제거 + 이동 마무리
- 퀵 토글 UI 및 `QuickState` 관련 코드 전부 삭제 (§7)
- Bug 2의 백업/복원 로직 삭제
- 기존 프로필 탭의 알림 설정 / 건강 프로필 수정 진입점 제거
- 알림 설정 → 설정 화면으로 이동
- 재진단 → 설정 화면으로 이동
- 설정 화면 아이템 전부 채움
- `/profile/edit` 라우트 삭제 + `/my-body-info` 리다이렉트 추가 (§Q5, §8.1)
- CareTab → `/my-body-info` 링크 변경 (§8.2)

## 10. 작업 원칙

### 10.1 절대 규칙

- **각 단계는 독립적으로 빌드되고 테스트 통과해야 함.** 중간 상태에서 앱이 깨지면 안 됨.
- **기능 추가 금지.** 리모델링에만 집중.
- **기존 코드 삭제는 해당 단계에서만.** 다음 단계에서 참조할 가능성이 있으면 유지.
- **UI 색상, 폰트, 간격 등 디자인 시스템은 기존 `design_tokens.dart` 따름.** 새 디자인 추가 금지.

### 10.2 놓치지 말 것

- 각 단계 작업 전, **영향받는 파일 목록**을 먼저 조사 후 보고
- 설계에서 빠뜨렸거나 모호한 부분이 있으면 **질문으로 멈추기**. 추측 금지.
- 각 단계 끝에 `flutter analyze` + `flutter test` 통과 필수
- 작업 단위가 크면 쪼개서 승인받기

### 10.3 질문 타이밍 (이럴 땐 꼭 멈춰서 물어보기)

- 설계 문서와 기존 코드가 충돌할 때
- "이 파일 건드려도 되나?" 애매할 때
- 설계에 명시되지 않은 선택지가 두 개 이상 있을 때
- 다른 feature (케어 탭, 리포트 탭 등)와 의존성이 보일 때
