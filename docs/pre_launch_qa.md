# Pre-Launch QA 체크리스트

## 디버그 빌드 설치 방법

### 1. 디버그 빌드로 연결된 폰에 설치

```bash
# USB 연결 후 기기 확인
flutter devices

# 특정 기기에 설치
flutter run --debug -d <device-id>

# 기기가 하나뿐이라면
flutter run --debug
```

> **참고**: `flutter run` 만 해도 기본값이 debug 모드야.

### 2. Crashlytics 디버그 수집 활성화 (단계 2 한정)

`main.dart`의 Firebase 초기화 직후에 아래 한 줄이 임시로 추가돼 있어:
```dart
await FirebaseCrashlytics.instance.setCrashlyticsCollectionEnabled(true);
```
이게 없으면 디버그 빌드에선 Crashlytics가 꺼져서 크래시가 Firebase에 올라오지 않아.

### 3. 설치 후 Crashlytics 수집 확인 방법

1. 앱 실행 후 **Firebase Console → Crashlytics → 해당 프로젝트** 접속
2. 좌측 "디버그" 탭 또는 "최근 이슈" 탭 확인
3. 터미널에서 `flutter logs` 또는 `adb logcat | grep -i crashlytics` 로 로컬 로그 확인
4. 수동 크래시 트리거 테스트:
   ```dart
   FirebaseCrashlytics.instance.crash(); // 임시 테스트용
   ```
   → Firebase 콘솔에 수 분 내 이슈 등장 확인

> **단계 3 전에**: `main.dart` 에서 `setCrashlyticsCollectionEnabled(true)` 라인과 TODO 주석을 반드시 제거할 것.

## 시나리오 1. 최초 실행 → 온보딩 완료

**대상 경로**: Splash → Tutorial → Roadmap → 온보딩(Q1~Q10) → 위치 설정 → 알림 시간 → 권한 → 완료 → Care 탭

- **정상 케이스**: 앱 최초 실행 시 튜토리얼이 표시되고, 질문 10개 모두 답변 후 위치를 자동 감지(또는 수동 선택)하면 홈(Care 탭)으로 이동한다.
- **비정상 케이스**: 온보딩 도중 앱을 강제 종료 후 재실행하면 온보딩 중간부터가 아닌 처음부터 다시 시작되어야 한다. (또는 이전 단계 유지 여부 확인)

## 시나리오 2. 위치 권한 거부 → 수동 지역 선택

**대상 경로**: LocationSetupScreen → 권한 거부 → 수동 선택 흐름

- **정상 케이스**: 위치 권한을 허용하면 현재 GPS 기준 가장 가까운 측정소가 자동 선택되고 다음 단계로 넘어간다.
- **비정상 케이스**: 위치 권한을 거부하면 수동 선택 UI가 표시되고, 영구 거부 시 "설정 열기" 버튼이 노출된다. GPS 타임아웃(20초) 발생 시에도 수동 선택 안내 메시지가 표시된다.
- [ ] 정상 통과
- [ ] 비정상 통과

## 시나리오 3. 알림 권한 거부 → 재요청 및 설정 안내

**대상 경로**: PermissionScreen → 거부 → 다이얼로그 → 설정 이동

- **정상 케이스**: 알림 권한을 허용하면 온보딩이 완료되고 Care 탭으로 이동한다. 이후 실제 알림이 정해진 시각에 발송된다.
- **비정상 케이스**: 거부 시 "설정에서 알림 허용" 다이얼로그가 뜨고, 영구 거부 시 "앱 설정 열기" 버튼이 동작한다. '건너뛰기' 선택 시에도 온보딩이 완료되고 홈으로 이동한다.
- [ ] 정상 통과
- [ ] 비정상 통과

## 시나리오 4. 홈(Care 탭) 네트워크 오류 → 재시도

**대상 경로**: CareTab → 먼지 데이터 fetch 실패 → 에러 위젯 → 재시도

- **정상 케이스**: 네트워크 정상 상태에서 홈 진입 시 현재 PM2.5, 등급, 마스크 권고, 예보가 표시된다. Pull-to-refresh로 최신 데이터를 불러온다.
- **비정상 케이스**: 네트워크를 끊은 상태에서 홈 진입 시 에러 위젯(재시도 버튼 포함)이 표시된다. 재시도 버튼 탭 후 네트워크 복구 시 데이터가 정상 표시된다.
- [ ] 정상 통과
- [ ] 비정상 통과

## 시나리오 5. 알림 탭 → "챙겼어요" 액션 → 방어 기록 저장

**대상 경로**: 알림 수신 → 액션 버튼 "챙겼어요" → SQLite 저장 → Report 탭 반영

- **정상 케이스**: 마스크 알림이 도착한 상태에서 "챙겼어요"를 탭하면 SQLite `notification_logs`에 `maskWorn` 액션이 기록되고, Report 탭의 방어율이 올라간다.
- **비정상 케이스**: 앱이 완전히 종료(killed state)된 상태에서 알림의 "챙겼어요"를 탭해도 동일하게 기록되어야 한다. 앱 재실행 후 Report 탭에 해당 기록이 반영되어야 한다.
- [ ] 정상 통과
- [ ] 비정상 통과

## 시나리오 6. 알림 탭 → "나중에 ✋" 스누즈 → 6시간 알림 억제

**대상 경로**: 알림 수신 → 스누즈 → snooze_until 저장 → 6시간 내 알림 미발송

- **정상 케이스**: "나중에 ✋"를 탭하면 `snooze_until` (현재 시각 + 6h)이 저장되고 모든 예약 알림이 취소된다. 6시간 내 백그라운드 dust check에서 예약 알림이 발송되지 않는다.
- **비정상 케이스**: 스누즈 중에도 PM2.5 ≥ 75 μg/m³ 긴급 알림은 발송되어야 한다. 6시간 경과 후엔 정상 알림이 재개된다.
- [ ] 정상 통과
- [ ] 비정상 통과

## 시나리오 7. 백그라운드 먼지 체크 → 알림 발송 검증

**대상 경로**: Workmanager `check_dust_task` (15분 간격) → 임계값 초과 → 알림 발송

- **정상 케이스**: 개인 T_final 초과 시 위험 진입 알림이 발송되고, SQLite에 로그가 기록된다. Firebase Analytics에 `notification_sent` 이벤트가 올라간다.
- **비정상 케이스**: 네트워크 오류 시 2회 재시도 후 캐시 fallback으로 동작하고 앱이 크래시되지 않는다. AirKorea 캐시도 없으면 알림을 건너뛰고 다음 주기에 재시도한다.
- [ ] 정상 통과
- [ ] 비정상 통과

## 시나리오 8. 마스크 착용 후 공기 개선 → 안전 진입 알림

**대상 경로**: 위험 진입 알림 → PM2.5 하강 → T_final + 15분 유지 → 안전 진입 알림

- **정상 케이스**: 위험 진입 알림 발송 후 PM2.5가 개인 임계값 이하로 15분 이상 유지되면 "공기가 맑아졌어요" 안전 진입 알림이 발송된다.
- **비정상 케이스**: 안전 진입 알림이 같은 날 중복 발송되지 않아야 한다. 스누즈 중에는 안전 진입 알림도 억제되어야 한다.
- [ ] 정상 통과
- [ ] 비정상 통과

## 시나리오 9. 프로필 Tier 2/3 변경 → 알림 임계값 즉시 반영

**대상 경로**: Profile 탭 → Tier 2(임신/피부시술) 또는 Tier 3(오늘 야외운동) 추가 → 다음 백그라운드 체크에서 강화된 기준 적용

- **정상 케이스**: Profile 탭에서 Tier 2(임신 상태)를 활성화하면 W_health 가중치가 높아져 T_final이 낮아진다. 이후 백그라운드 체크에서 새로운 T_final 기준으로 알림이 발송된다.
- **비정상 케이스**: Tier 2 피부시술을 14일 전 날짜로 설정하면 자동 만료되어 Tier 2 가중치가 제거된다. Tier 3는 당일 자정에 자동 리셋된다.
- [ ] 정상 통과
- [ ] 비정상 통과

## 시나리오 10. 앱 재설치 후 온보딩 초기화 확인

**대상 경로**: 앱 삭제 → 재설치 → 신규 사용자 플로우 시작

- **정상 케이스**: 앱 재설치 후 실행 시 튜토리얼부터 다시 시작한다. 이전 UUID는 삭제되고 새로운 익명 사용자 ID가 생성된다. 이전 SQLite 데이터(aqi_records, notification_logs)가 없는 상태에서 24시간 초기 데이터 시딩이 실행된다.
- **비정상 케이스**: SharedPreferences가 남아 있는 상태(재설치가 아닌 강제 데이터 삭제)에서 `prefOnboardingCompleted`가 true이면 온보딩 없이 홈으로 이동하고 SQLite가 빈 상태여도 앱이 크래시되지 않아야 한다.
- [ ] 정상 통과
- [ ] 비정상 통과
