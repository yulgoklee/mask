## 이 문서의 목적

`sensitivity_signals_v0.md`의 48개 잠재 민감군 신호를 mask_alert의 `T_final` 계산식 (`ThresholdConfig` / `ThresholdEngine`) 위에 올리기 위한 **가중치 매핑 초안**.

> **중요 — 적용 전 전제 조건**
> - yulgok이 자료원 16개 URL 검증 완료해야 함 (현재 미완)
> - 검증 통과 못 한 자료원이 인용된 신호는 가중치 0으로 폐기
> - 본 문서는 "검증 통과 후 적용" 전제. 검증 전에 코드 변경 금지.
> - **의료법 준수**: "민감군 가능성"으로만 표현. "당신은 천식입니다" 같은 진단 표현 절대 금지.

## 0. 기존 시스템 요약 (참조)

### 0.1 현재 가중치 (`ThresholdConfig.defaults`)

```
T_base = 35   (환경부 '나쁨' 진입)
T_floor = 15  (WHO 2021 단기 권장)

T_final = clamp(T_base × (1 - W_age - W_health), T_floor, T_base)
W_health = W_respiratory + W_cardiovascular + W_smoking
```

|---------|----|-------|------------|
| 호흡기 | asthma (천식) | 0.20 | **0.30** |
| 호흡기 | copd (COPD) | 0.25 | 0.30 |
| 호흡기 | rhinitis (비염) | 0.15 | 0.30 |
| 호흡기 | allergy (알레르기) | 0.15 | 0.30 |

| 심혈관 | heartDisease (심장) | 0.20 | 0.25 |
| 심혈관 | stroke (뇌졸중) | 0.15 | 0.25 |

### 0.2 신호 강도 정의 (catalog)
- **강 (strong)**: 의학 검사·임상 기준 직접 시사, 확정에 가까움
- **중 (medium)**: 시사하지만 다른 원인 배제 못함, 통계적 약함
- **약 (weak)**: 노출 표지·간접 신호, 단독 판단 어려움

## 1. 매핑 원칙 (Mapping Principles)

설계 의사결정을 명시해서 향후 일관성·재현성 확보.

### 1.1 신호는 "진단명의 분수(fraction)"로 환산
- 신호가 진단명을 직접 시사하면, 그 진단명 가중치의 일부를 부여한다.
- 예: 비염 진단 = 0.15 → 비염 신호 1개 (강) ≈ 0.15 × 1/3 = 0.05
- 신호 여러 개가 합쳐져도 진단명 가중치를 넘지 못하게 한다 (cap에서 처리).

### 1.2 신호 강도별 기본 비율 (개별 진단명 가중치 대비)
| 강도 | 비율 | 환산 예 (진단명 0.15 기준) |
|-----|------|--------------------------|
| 강 | 약 1/3 | 0.05 |
| 중 | 약 1/5 | 0.03 |
| 약 | 약 1/8 | 0.02 |

진단명 가중치가 0.20·0.25라면 비례해서 0.07·0.08 등. 0.01 단위 반올림.

### 1.3 보수적 설정
- 의심 가면 작게. 신호는 자가 보고라 noise 큼.
- yulgok 검증 안 끝났으면 모두 "yulgok 검증 필요" 마킹.
- Phase 3 사용자 데이터로 보정 후에 정식 값 결정.

### 1.4 이중 카운팅 방지 (Double-counting)
**가장 중요한 원칙.** 진단명과 신호가 같은 사람에게서 동시에 잡히면 가중치 inflation 위험.

규칙:
- **R1 (진단명 우선)**: 진단명이 이미 체크되어 있으면, 같은 카테고리의 신호 가중치는 **0으로 한다**.
- **R2 (카테고리 cap이 최종 안전망)**: R1을 빠뜨려도 카테고리 cap이 inflation 막음.
- **R3 (교차 카테고리 신호)**: 신호가 여러 카테고리에 걸치면 **상위 강도 카테고리 1곳에만** 부여.

| 상황 | 처리 |
|-----|------|
| `rhinitis=true` + 신호 A1·A2·A3 모두 yes | A1·A2·A3 가중치 모두 0 (R1) |
| `asthma=false`, 신호 B1·B2·B3 모두 yes | 합산하되 respiratoryCap 0.30 안에서 처리 |
| 신호 D7 (흡연 + 호흡기) yes, `smoking_current=true` | 흡연 가중치는 진단명 트랙으로 처리, D7은 호흡기 신호로만 부여 (강도 약하게) |

### 1.5 카테고리 cap은 그대로 유지 (신호 합산도 cap 안에서)
- 호흡기 cap 0.30, 심혈관 cap 0.25는 **변경하지 않는다**.
- 진단명 + 신호 둘 다 호흡기 cap 0.30 안에서 합산되도록.
- 이유: cap을 올리면 T_final이 floor(15)에 너무 빨리 도달해서 사용자가 항상 "위험" 알림 받게 됨 → 알림 피로도 ↑
- Phase 3에서 검증 통과 신호만 남기면 자연스럽게 cap에 잘 맞을 것.

## 2. 신호별 가중치 매핑 (48개)

> **표 읽는 법**
> - "가중치(검증 후)" — yulgok 자료원 검증 통과 시 적용. 통과 못 하면 0.
> - "이중 카운팅 룰" — 어떤 진단명이 켜져 있을 때 이 신호 가중치를 0으로 만들지 명시.

## 3. 카테고리 cap 정책 결정

### 3.1 결론 요약
| 항목 | 결정 |
|-----|------|
| 호흡기 cap (0.30) | **유지** — 진단명 + A·B·C·D·E·G·H 신호 전부 합산 |
| 심혈관 cap (0.25) | **유지** — 진단명 + F 신호 합산 |
| 새 cap 신설 | **하지 않음** |
| 흡연 가중치 | 별도 트랙 유지 (cap 없음, §4) |

### 3.2 EIB (C) — 호흡기 통합

선택지:
- **(a) 호흡기 cap에 통합** ← 채택
- (b) 별도 cap (예: 0.10)

근거:
- EIB는 천식의 변형으로 ATS·KAAACI에서 호흡기 질환으로 분류 (S2, S15)
- 별도 cap 만들면 가중치 inflation 위험 (cap 0.30 + 0.10 = 0.40 → floor 도달 빨라짐)
- C 그룹 단독 max 0.26은 호흡기 cap 0.30 안에서 자연스럽게 처리됨

### 3.3 만성 기관지염 (D), 수면 무호흡 (G), 일반 호흡기 (H) — 호흡기 통합

선택지:
- **(a) 호흡기 cap에 통합** ← 채택
- (b) 별도 cap

근거:
- 만성 기관지염 (D)은 GOLD 정의상 COPD 일종 (S6) → 동일 카테고리 자연스러움
- 수면 무호흡 (G)는 호흡기 영향이 우세 (저산소 → 폐 부담)
- 일반 호흡기 (H)는 WHO 호흡기 영향 (S3)
- cap 분리하면 사용자 경험에서 임계치 너무 빨리 floor에 도달

### 3.4 심혈관 (F) — cardiovascularCap에 통합

근거: AHA/Framingham 모두 심혈관 위험 통합 평가 (S8). cap 0.25 유지.

### 3.5 cap 검증: 신호만 켰을 때 최악 시나리오

진단명 모두 false라고 가정 (즉, 사용자가 진단 안 받았지만 신호는 다 켜진 잠재 민감군).

| 그룹 | 단독 max | cap 적용 후 |
|-----|---------|-----------|
| 호흡기 (A+B+C+D+E+G+H 합산) | 0.36 + 0.56 + 0.26 + 0.44 + 0.14 + 0.09 + 0.09 = **1.94** | **0.30** (cap) |
| 심혈관 (F) | 0.35 | **0.25** (cap) |

→ cap이 효과적으로 작동. 합산 inflation 안 일어남.

신호만으로 호흡기 cap 도달하는 사용자는 진단 받은 사람과 동급으로 보호됨 → **잠재 민감군 발견 효과 달성**.

## 4. 흡연 신호 처리 (D7 vs smoking_current/former)

### 4.1 현재 시스템
- `smoking_current=true` → +0.20 (cap 없음)
- `smoking_former=true` → +0.10 (cap 없음)
- 진단명·흡연은 별도 트랙 (W_smoking)

### 4.2 D7 신호 (흡연력 + 호흡 증상)
의미: 단순 흡연자 중에서도 **호흡 증상이 동반**된 사람 → COPD 잠재군. 흡연 자체가 아닌 **상호작용 신호**.

처리 방침:
- D7은 **호흡기 카테고리** 신호로 +0.05 부여 (D 표 참조)
- 흡연 자체는 별도 W_smoking에서 이미 잡힘 (중복 계산 안 함 — D7은 "증상 + 흡연력"이지 "흡연력 그 자체"가 아님)
- `smoking_current=true` AND D7=true → W_smoking 0.20 + W_resp +0.05 = 0.25 (서로 다른 트랙이라 합산 OK)

### 4.3 smoking 자체에 cap 도입 검토 → 보류
- 현재 cap 없음. W_smoking 최대 0.20.
- 신호 추가로 cap 도입할 정도는 아님. 0.20은 단일 항목이라 inflation 없음.
- yulgok 결정 사항으로 남김. 권장: 현 상태 유지.

## 6. Trade-off 분석

### 6.1 신호 수 vs 사용자 부담

| 신호 수 | 장점 | 단점 |
|--------|-----|-----|
| 0개 (현 상태) | 빠른 온보딩, 안정 | 잠재 민감군 못 잡음 |
| **5~7개 (권장)** | 핵심 잠재군 발견, 부담 적음 | 일부 강 신호 누락 |
| 10~15개 | 정밀도 ↑ | 온보딩 길어짐, 이탈률 ↑ |
| 30개+ | 학술적 완성도 | 사용자 이탈, 알림 inflation |

### 6.2 가중치 inflation 위험
- 신호 7개 모두 호흡기 가중치 합산 max 0.52 → cap 0.30이 잘 막아줌
- 진단명 + 신호 동시 켜면 R1 (이중 카운팅 방지)이 잡아줌
- 검증 안 된 단계에서는 cap 0.30 유지가 안전

### 6.3 사용자 혼란
- 의료법 준수: "당신은 천식일 가능성" 같은 표현 X. 대신 "민감군에 해당할 수 있어 임계치를 낮게 설정합니다" 같은 중립 표현

- **진단명 자동 매핑 (B7→`asthma=true`, F1→`hypertension=true`) 검토** — Phase 4에서 결정

### 6.4 권장 도입 신호 수: **사이클 #8 알파에서 7개**

## 7. 검증 후 적용 절차 (Phase 3)

### 7.1 데이터 수집 (출시 + 28~84일)

알파 사용자에게 추가 7개 신호 답변 수집. SQLite (`local_database.dart`)에 신호 답변·시점·해당 사용자 알림 액션 같이 저장.

스키마 예시 (Phase 4 코드 작업 시 참고):
```sql
-- (실제 작업은 코드 변경 단계에서. 이 문서는 매핑 초안만)
signal_answers (
  user_id TEXT,
  signal_id TEXT,         -- 'A1', 'B7' 등
  answer BOOLEAN,
  answered_at INTEGER
)
```

### 7.2 분석 지표

#### 7.2.1 1차 분석 — 신호별 응답 분포
- 각 신호 yes/no 비율
- yes 비율 < 5% 또는 > 80% → 분별력 낮음 (제외 후보)

#### 7.2.2 2차 분석 — 신호 ↔ 행동 상관
타깃 행동 (proxy outcome):
- 알림 탭 후 마스크 착용 액션 (`LocalDatabase.updateUserAction`)
- 앱 재실행 빈도 (장기 사용)
- 알림 ON 유지율 (4주 후)

분석 방법:
- **Pearson 상관계수** (신호 yes/no × 액션 횟수): 0.1 미만이면 약함
- **이항 로지스틱 회귀**: 신호 → "마스크 착용 액션 1회 이상" 예측. p < 0.05 + odds ratio > 1.5
- **t-test**: 신호 yes 그룹 vs no 그룹의 액션 평균 차이 (p < 0.05)
- **카이제곱**: 신호 yes/no × 알림 ON 유지율 분포

#### 7.2.3 3차 분석 — 신호 간 상관 (다중공선성)
- 두 신호 간 상관 > 0.7 → 둘 중 하나만 남김 (예: A1·A2 둘 중 하나)
- VIF (Variance Inflation Factor) > 5 주의

### 7.3 가중치 보정 (Phase 3 산출물)
- 신호별 검증 통과/탈락 결정
- 통과 신호의 가중치를 actual odds ratio 기반으로 재조정
- `signal_weight_mapping_v0.1.md` 작성

### 7.4 통합 (Phase 4)
- 통과 신호로 `ThresholdConfig`·`ThresholdEngine` 확장
- `HealthSignalEntry` 신규 클래스 (가중치 + 이중 카운팅 룰 키 포함)
- 온보딩 신규 페이지 추가 (현재 Q4/Q5 뒤)
- 자기 점검 기능 (사이클 #9 or #10)
