## 1. 왜 재설계하는가

### 1.1 현재 상태 진단 (스크린샷 기반)

신규 유저가 보는 10개 화면 흐름:
```
splash → tutorial(2장) → roadmap → onboarding(8~9장) → analysis_loading
→ dashboard → location_setup → notification_time → permission → complete
```

**문제 1 — 가중치 노출 (가장 심각)**

Q4 호흡기, Q5 민감도, Q7 활동량, Q8 마스크 불편도 모든 화면에 `+0%`, `+5%`, `+15%`, `+30%`, `-10%` 같은 가중치 배지가 보임.

```
😊 건강해요 [+0%]
🐽 비염 있어요 [+15%]
🫁 천식 등 질환 [+30%]
```

이건 개발자가 보는 내부 가중치이지, 유저에게 노출할 정보가 아님. 케어 탭이 "프로페셔널한 친구" 톤을 추구하는데 온보딩에서 유저에게 "당신은 +30% 위험"이라고 라벨링하는 건 정반대 무드.

대시보드 "민감도 기여 항목"에서도 동일한 가중치 막대가 노출됨.

**문제 2 — 대시보드가 옛 디자인**

분석 로딩 후 "yulgok님의 기준이 만들어졌어요" 화면(`/dashboard`)이 새 프로필 탭과 완전히 다른 디자인:
- 페르소나 이름이 옛것 ("기본 관리형" 등 — 우리가 6종 새로 만든 것과 무관)
- "위험도 구간 차트" — 5단계 색상 영역 차트, 케어 탭 어디에도 없음
- "민감도 기여 항목" — 가중치 막대 그래프, 새 페르소나 시스템 reasons과 중복

신규 유저가 보는 첫 진단 결과 화면이 매일 보게 될 프로필 탭과 따로 놂.

**문제 3 — 로드맵이 거짓말**

"딱 3분이면 돼요. Step 1~3"으로 약속하지만 실제는 [진단 → 분석 → 대시보드 → 위치 → 알림 → 권한 → 완료] 7단계. 유저가 알림 설정 후 끝나는 줄 알았는데 권한 화면이 또 나옴.

**문제 4 — 튜토리얼 + 로드맵이 분리**

앱 소개 정보가 두 화면(`/tutorial` 2장 + `/roadmap` 1장)에 분산. 진짜 시작(Q1)까지 화면 3번 거쳐야 함.

**문제 5 — 진행 표시 "나중에" 모호**

`1 / 9 나중에` — "나중에"가 건너뛰기 버튼인지, 진행 라벨인지 의미 불명.

**문제 6 — 알림 문체 선택의 정체성 모호**

알림 시간 화면 하단에 "다정한 가디언 vs 단호한 분석가" 선택. 그러나 케어 탭은 이미 "프로페셔널한 친구" 톤으로 고정. 이 선택이 실제로 뭘 바꾸는지 불분명. 설령 알림 메시지만 바뀐다 해도 케어 탭/알림 톤 분리는 모순.

### 1.2 재설계 방향 (한 문장)

> **신규 유저의 첫 진단을 케어 탭/프로필 탭과 동일한 무드("프로페셔널한 친구")로 통일하고, 가중치 같은 개발자 정보는 완전히 숨긴다.**

### 1.3 온보딩의 본질

**유저 관점에서 온보딩은 "이 앱이 나를 알게 만드는 시간"이다.**

- 빠르게 끝나는 게 미덕이 아님 → **신뢰감 있게 끝나는 것**이 미덕
- 짧다고 좋은 게 아님 → 유저가 **"내 정보를 정확히 받았구나"** 느끼면 길어도 OK
- 단, **"내가 뭘 답했는지" 명확**해야 함 — 가중치 % 노출은 정확함이 아니라 위협

**유저가 마지막에 받아야 할 감정**: "이 앱이 나를 이해했다."
가중치 라벨링은 정반대 효과 — "이 앱이 나를 위험인물로 분류했다."

## 2. 새 온보딩 흐름 (8단계)

```
1. /splash              스플래시 (변경 없음)
   ↓
2. /welcome             환영 (tutorial+roadmap 통합) — 신규
   ↓
3. /onboarding          진단 (Q1~Q8, 가중치 제거)
   ↓
4. /analysis_loading    분석 로딩 (변경 없음)
   ↓
5. /diagnosis_result    진단 결과 (dashboard 대체) — 새 페르소나 시스템 활용
   ↓
6. /location_setup      위치 설정 (변경 없음)
   ↓
7. /notification_time   알림 시간 (알림 문체 선택만 제거)
   ↓
8. /onboarding_complete 완료 (변경 없음)
```

기존 10단계 → 8단계. 단, **압축이 핵심이 아니라 통합과 일관성**이 핵심.

`/permission` 라우트는 분기 시점 재검토 필요 (단계 0 조사에서 결정).

- 단계 안내: 13px Medium 라벨 + 14px Regular 설명
- "시작할게요" CTA: 기존 AppButton 패턴

### 3.2 온보딩 진단 (/onboarding) — 가중치 제거

#### Q1~Q8 공통 변경

**(a) 가중치 배지 완전 제거**

```
기존:                          이후:
😊 건강해요 [+0%]              😊 건강해요
              호흡기 관련 증상이 없어요
🐽 비염 있어요 [+15%]          🐽 비염
              코막힘·재채기가 자주 발생해요
🫁 천식 등 질환 [+30%]         🫁 천식 등 질환
              호흡기·심혈관 질환을 진단받았어요
```

배지 자리를 비워둠 (체크 아이콘 정도만 선택 표시).

**(b) 안내 문구 (💡 박스) 톤 정비**

```
기존: "비염이 있으면 +15%, 천식 등 호흡기 질환이 있으면
      +30% 강화돼요. 두 가지 모두 해당되면 중복 적용
      (+45%)되어 더 일찍 알려드려요."

이후: "비염이 있으시면 코 점막이 먼지에 민감해요.
      천식이 있으시면 적은 농도에도 기관지가 반응해요.
      당신의 상태에 맞춰 더 일찍 알려드릴게요."
```

**원칙**:
- 숫자 (+%, ±) 일체 제거
- 케어/프로필 탭의 reasons 톤과 동일 (공감 + 팩트 + 끝맺음)
- 의학적 근거가 있는 경우 그대로 유지 (단, 가중치 숫자 빼고)

**(c) 진행 표시 "나중에" 처리**

`1 / 9 나중에` → `1 / 8` 만 표시. "나중에"는 건너뛰기 버튼이라면 별도 위치로 이동.

조사 단계에서 "나중에"가 무엇인지 명확히 확인 후 결정:
- 건너뛰기 버튼이면 → 우측 상단 별도 텍스트 버튼으로 분리
- 진행 라벨이면 → 그냥 삭제

#### 질문별 카피 정비 (참고용)

기존 카피 톤은 대체로 좋음. 가중치 노출 제거 + 안내 박스 톤 정비만 핵심.

**Q1 닉네임**: 그대로 (좋음)
**Q2 출생연도**: 안내 박스에서 "10% 추가 강화돼요" → "기준치를 자동으로 조정해드려요"
**Q3 성별**: 그대로
**Q4 호흡기**: 가중치 + 안내 정비
**Q5 민감도**: 가중치 + 안내 정비
**Q6 피부 시술**: 그대로 (가중치 노출 없음)
**Q7 활동량**: 가중치 + 안내 정비. "흡입량이 최대 3배 증가" → 보존 (의학적 사실, 가중치 아님)
**Q8 마스크 불편도**: 가중치 + 안내 정비

### 3.3 진단 결과 (/diagnosis_result) — 신규 (기존 /dashboard 대체)

**가장 큰 변경.** 새 페르소나 시스템과 통합.

#### 핵심 변경

기존 `/dashboard` 화면의 다음 요소 **삭제**:
- "민감도 기여 항목" 가중치 막대
- "위험도 구간 차트" (5색 영역 차트)
- "감정 섹션" (현재 위에 있는 것)

신규: 프로필 탭의 페르소나 카드 + 진단 톤 결합.

#### 레이아웃

```
┌─────────────────────────────────┐
│                                 │
│  yulgok님,                      │   ← 인사
│  이렇게 알려드릴게요.            │
│                                 │
│  ┌───────────────────────────┐ │
│  │ [페르소나 이모지]          │ │
│  │ [페르소나 이름]            │ │   ← 새 6종 페르소나
│  │ [서브타이틀]               │ │   (균형 유지형 등)
│  │                            │ │
│  │ 내 기준치    19 µg/m³      │ │
│  │ 일반인 기준  35 µg/m³      │ │
│  │                            │ │
│  │ ─────────                  │ │
│  │                            │ │
│  │ 왜 이 기준인가요           │ │   ← reasons 그대로
│  │                            │ │
│  │ ① [근거 1]                 │ │
│  │ ② [근거 2]                 │ │
│  │ ③ [근거 3]                 │ │
│  └───────────────────────────┘ │
│                                 │
│  공기가 나빠지기 전에            │   ← 안심 메시지

```

#### 차이점 (프로필 탭 페르소나 카드와)

- **자세히 보기 / 접기 토글 없음** — 진단 결과는 처음부터 펼친 상태
- 카드 위에 인사 메시지 추가 ("yulgok님, 이렇게 알려드릴게요")
- 카드 아래에 안심 메시지 추가 ("공기가 나빠지기 전에 먼저 알려드릴게요")
- CTA 버튼 추가 ("위치 설정으로")

#### 균형 유지형 처리

`general` 페르소나 (균형 유지형)는 reasons가 빈 리스트.

레이아웃:
```
[페르소나 카드]
  [균형 유지형]
  [지금 상태라면 기본 기준으로 충분해요]
  
  내 기준치    35 µg/m³
  일반인 기준  35 µg/m³
  
  ─────────
  
  지금 상태라면 공식 기준을 그대로
  적용해도 충분해요.
```

#### 시각 스펙

- 카드 모서리: 16px (프로필 탭과 동일)
- 카드 내부 padding: 24px
- 페르소나 배경색: 6종 매핑 그대로 (프로필 탭과 동일)
- 페이지 패딩: horizontal 20px

#### 데이터 흐름

```
profileProvider → PersonaGenerator.generate(profile) → Persona
  - persona.type, emoji, name, subtitle
  - persona.reasons (List<ReasonItem>)
  - profile.tFinal
```

프로필 탭 페르소나 카드 위젯의 **확장 상태 부분만** 재사용.

### 3.4 알림 시간 (/notification_time) — 알림 문체 선택 제거

#### 현재 → 이후

```
[현재]                          [이후]
외출 전 알림 카드               외출 전 알림 카드
전날 예보 알림 카드             전날 예보 알림 카드
귀가 후 알림 카드               귀가 후 알림 카드
                                
알림 문체                       (제거)
  다정한 가디언 vs 단호한 분석가
                                
[알림 미리 받아보기]            [알림 미리 받아보기]
[설정 완료]                     [설정 완료]
```

#### 후속 처리

- 기존 알림 문체 데이터 (예: `notificationTone` 같은 필드)가 어디서 참조되는지 조사
- 케어 탭/알림에서 이 필드 사용처 발견 시 → 분기 코드 제거 + "프로페셔널한 친구" 단일 톤으로 통합
- 기존 사용자의 저장된 값은 무시 (마이그레이션 불필요)

권한 화면이 **신규 유저에게 맥락 있게 권한을 설명**하는 역할이라면 유지. 단순 권한 팝업이라면 분기 재검토.

## 4. 시공 단계 (4단계)

### 단계 0: 가중치 노출 완전 제거 (최우선)

UI에 노출되는 모든 가중치 배지/숫자 제거. 안내 카피 톤 정비.

**범위**:
- Q4, Q5, Q7, Q8의 가중치 배지 제거
- 모든 안내 박스(💡)의 +% 표현 정비
- 진행 표시 "나중에" 처리
- 대시보드는 이번 단계에서 건드리지 않음 (단계 1)

**테스트**: 카피 매트릭스 검증, 가중치 노출 없음 확인 (grep 기반).

### 단계 1: 진단 결과 화면 (/diagnosis_result) 신규

기존 `/dashboard` 화면을 대체.

**범위**:
- 새 화면 파일 생성 (`lib/features/onboarding/diagnosis_result_screen.dart`)
- 프로필 탭의 페르소나 카드 위젯 재활용 (확장 상태 고정)
- 라우터 수정: `/dashboard` → `/diagnosis_result` 변경 또는 리다이렉트
- 기존 `/dashboard` 화면 파일은 유지 (단계 5에서 삭제 검토)
- 위험도 구간 차트, 민감도 기여 항목 등 제거된 위젯들 정리

**테스트**: 페르소나 6종 각각 결과 렌더링, 균형 유지형 빈 reasons 처리.

### 단계 2: 환영 화면 통합 (튜토리얼+로드맵 → /welcome)

**범위**:
- 새 화면 파일 생성 (`lib/features/onboarding/welcome_screen.dart`)
- 라우터에 `/welcome` 라우트 추가
- 스플래시 → `/welcome` 으로 라우팅 (기존 `/tutorial` 대신)
- 기존 `/tutorial`, `/roadmap` 라우트는 유지 (단계 5에서 삭제 검토)

**테스트**: 진입 가능, "시작할게요" 버튼 → `/onboarding` 라우팅.

### 단계 3: 알림 문체 제거 + 정리

**범위**:
- `/notification_time` 화면에서 알림 문체 선택 UI 제거
- `notificationTone` (또는 비슷한 필드) 사용처 grep
- 사용처에서 분기 제거, "프로페셔널한 친구" 단일 톤으로 통합
- 기존 SharedPreferences 키는 그대로 둠 (마이그레이션 불필요)

**테스트**: 화면 렌더링, 톤 분기 코드 제거 확인.

### 단계 4: 최종 점검 (선택적)

**범위**:
- 단계 0~3 통합 동작 확인
- 신규 유저 흐름 8단계 전체 시뮬레이션
- 더 이상 사용 안 되는 화면 파일 삭제 (`/tutorial`, `/roadmap`, `/dashboard`)
- 단, 이번 출시 후 1~2주 관찰 후 안전 삭제 권장 → 단계 4는 출시 후 작업으로 보류 가능

## 5. 작업 원칙

### 5.1 절대 규칙

- **각 단계는 독립적으로 빌드되고 테스트 통과해야 함.**
- **기존 화면 파일 삭제 금지** (단계 4 이전까지). 라우팅만 끊음.
- **단계 0 (가중치 제거) 완료 없이 단계 1, 2 진입 가능** (독립적이므로).
- **새 디자인 토큰 추가 금지.** 기존 DT 토큰만 사용.
- **새 기능 추가 금지.**
- **유저 데이터 형식 변경 금지** (UserProfile 등).

### 5.2 놓치지 말 것

- 각 단계 작업 전, **영향받는 파일 목록**을 먼저 조사 후 보고
- 설계 모호 시 **질문으로 멈추기**. 추측 금지.
- 각 단계 끝에 `flutter analyze` + `flutter test` 통과 필수
- 톤 일관성: 케어 탭/프로필 탭의 카피 무드를 따름

### 5.3 질문 타이밍

- 설계 문서와 기존 코드가 충돌할 때
- "나중에" 라벨이 무엇인지, 알림 문체가 어디서 쓰이는지 등 **불명 사항 발견 시**
- 다른 feature에 영향 예상 시

## 6. 테스트 전략

### 단계 0 테스트

- 모든 온보딩 화면에서 가중치 배지 노출 없음 (위젯 트리 검색)
- 안내 박스 카피 매트릭스 검증
- 진행 표시 형식 (`1 / 8`) 검증

### 단계 1 테스트

- 6개 페르소나 각각 진단 결과 화면 렌더링
- 균형 유지형 빈 reasons 처리 (대신 안내 문구 표시)
- "위치 설정으로" CTA 동작
- 인사/안심 메시지 카피 표시

### 단계 2 테스트

- 환영 화면 진입 후 "시작할게요" → `/onboarding` 이동
- 스플래시 → `/welcome` 라우팅 (기존 `/tutorial` 대신)

### 단계 3 테스트

- 알림 시간 화면에서 알림 문체 선택 UI 없음
- 케어 탭/알림 톤 분기 제거 후 정상 동작
